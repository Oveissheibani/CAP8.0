# CAP 8.0 — Codebase Map

A complete tour of the **Correlation Analysis Package (CAP)** — a ROOT-based C++ framework for analyzing particle-physics events (correlations, balance functions, jets, flow, HBT, etc.). Generated by mapping every directory of `/Users/oveissheibani/cap8/CAP8.0-main`.

---

## 1. What CAP Is

CAP is a **plugin-style analysis framework** for high-energy / heavy-ion physics. You write a `.ini` configuration file describing a tree of analysis stages; the framework instantiates the corresponding C++ classes by name (via ROOT's reflection system), wires them together, and runs the lifecycle. A typical job ingests events from a Monte-Carlo generator (Pythia 8, Therminator 3, EPOS, PHSD, Glauber, or a basic toy), filters them, fills ROOT histograms, then post-processes those histograms into derived observables (cumulants, balance functions, flow harmonics, HBT correlations, jet observables…).

**Core dependencies**
- **ROOT** (CERN) — histograms, I/O, dictionaries, reflection
- **PYTHIA 8** — optional, for `CAPPythia` module
- **FastJet 3.4.3** — optional, vendored at `fastjet-3.4.3/`, used by `Jets/`

## 3. Build & Run Pipeline

### 3.1 Building

`SetupCAP.sh` exports `CAP_*` environment variables (paths to source, build, projects, libs, plus `CAP_PYTHIA8_*` and `CAP_FASTJET_*`). Then:

```bash
source SetupCAP.sh
mkdir build && cd build
cmake ../src
make -j8 install
```

`src/CMakeLists.txt` is the master build script. Optional modules are gated by flags at the top:

```cmake
set(Include_JETS "NO")        # FastJet-based jet reconstruction
set(Include_PYTHIA "NO")      # Pythia 8 generator
set(Include_HERWIG "NO")
set(Include_HIJING "NO")
set(Include_AMPT "NO")
set(Include_EPOS "NO")
set(Include_PHSD "NO")
set(Include_THERMINATOR "YES")
set(Include_GLAUBER "YES")
set(Include_IDENTITY "NO")
```

Each `src/<Module>/` subdir has its own `CMakeLists.txt` that:
1. Calls `ROOT_GENERATE_DICTIONARY` on the module's `*LinkDef.h` to emit a ROOT dictionary
2. Compiles the `.cpp` files into a shared library (`libBase.so`, `libParticles.so`, …)
3. Installs to `lib/` so `CAP.cxx` and ROOT macros can `dlopen` them

The final executable `CAP` links Base, Math, Glauber, Particles, SubSample, Global, Spherocity, ParticleSingle/Pair/Pair3D, NuDyn, PtPt, Performance, Therminator3, BasicEventGen, Xml, Latex (Pythia / Jets / EPOS / PHSD added when their flags are `YES`).

### 3.2 Running

```bash
./CAP TaskName projectPath configFile.ini histogramOutPath
```

`CAP.cxx` does:

1. Parse argv → `taskName`, `configurationPath`, `configurationFile`, `histogramPath`
2. Set the ROOT random seed
3. `CAP::Configuration configuration; configuration.loadFrom(path, file);` — flatten the `.ini` into a `Properties` map
4. Create a "creator" `Task` that calls `createTask(nullptr, taskName, configuration)` — this is the recursive factory
5. `mainTask->initialize()` → `mainTask->execute()` → `mainTask->finalize()` → delete

A single big `try/catch` block catches any of CAP's domain-specific exceptions (`ConfigurationException`, `MathException`, `FileException`, `MemoryException`, `HistogramException`, `FactoryException`, generic `Exception`) and prints a diagnostic.

### 3.3 The .ini language

Configuration files are flat, colon-delimited, hierarchical key / value pairs:

```ini
RunAnalysis:TaskName              = RunAnalysis
RunAnalysis:TaskClassName         = CAP::RunAnalysis
RunAnalysis:nSubtasks             = 4
RunAnalysis:Subtask3:TaskName     = EventIterator
RunAnalysis:Subtask3:TaskClassName = CAP::EventIterator
RunAnalysis:Subtask3:Subtask0:TaskName        = PythiaEventGenerator
RunAnalysis:Subtask3:Subtask0:TaskClassName   = CAP::PythiaEventGenerator
…
```

The path encodes the parent → child task tree. Each task block declares:
- `TaskClassName` — the C++ class to instantiate (looked up via `TClass::GetClass`)
- `nSubtasks`, `Subtask<i>:*` — children
- `nParticleDbs`, `nEventFilters`, `nParticleFilters`, `nJetFilters`, `nEventsStreams` — resource counts and names
- Per-task flags (`ImportHistograms`, `ExportHistograms`, `Severity`, …)
- Algorithm-specific parameters (binning ranges, filter conditions, Pythia options, …)

A second class of blocks (e.g. `EventFilterCreator:EventFilter:Filter1:Condition0:Type = DoubleRange`) defines reusable filter templates that the creator tasks instantiate by name.

## 4. The Base/ Framework — How Everything Hangs Together

### 4.1 The Task abstraction

`CAP::Task` (in `src/Base/Task.hpp`) is the central abstraction. It inherits from both `Object` (name/title) and `ConfigurationManager` (config state).

Lifecycle (each is virtual; default impls iterate over children):

| Method                | Purpose                                                       |
| --------------------- | ------------------------------------------------------------- |
| `setDefaultConfiguration()` | Hard-coded fallback values                              |
| `configure()`         | Pull values from `_configuration`                             |
| `initialize()`        | Build histograms, open files, allocate                        |
| `execute()`           | Per-event work (or per-job for top-level tasks)               |
| `partial()`           | Periodic save (e.g. every N events)                           |
| `finalize()`          | Write histograms, close files, free resources                 |
| `reset()`, `clear()`  | Reset counters / state                                        |

Each Task carries:
- `_parentTask`, `_subTasks` — tree pointers
- `_configuration` — its slice of the global `Properties`
- Lists of `EventFilter*`, `ParticleFilter*`, `JetFilter*`, `ParticleDb*`, `EventStream*` — owned or borrowed (the `Owner` flag in the .ini decides)
- An `Accountant` `_taskExecuted` (event counter) and a `Timer`

### 4.2 The factory

`Task::createTask(parent, name, config)` reads `name:TaskClassName`, then uses ROOT reflection:

```cpp
TClass * cls = TClass::GetClass(taskClassName);
Task * t = static_cast<Task*>(cls->New());
```

This is why every Task subclass needs (a) a default constructor and (b) an entry in its module's `*LinkDef.h` so the ROOT dictionary registers it. Failure throws `UnknownClassException`.

### 4.3 Configuration system

- `Property` — name + title + value (stored as `TString`, with typed accessors `integer()`, `doubleValue()`, `isBool()`)
- `Properties` — `std::vector<Property>` with path-based key lookup
- `Configuration` — adds templated helpers (`generateKeyValuePairs`, `selectedValues`)
- `ConfigurationManager` — mixin owning a `Configuration`, exposing `valueInt(path,key)`, `valueBool`, `valueDouble`, `valueString`
- `PropertiesParser` (extends `Parser`) — parses / serializes `.ini` files

### 4.4 Filters

`Filter<T>` (Filter.hpp) is a template holding a `vector<Condition*>`. The base template throws `FilterException` on `accept(const T&)`; concrete instantiations provide logic.

Conditions are polymorphic:
- `ConditionBool`, `ConditionInteger`, `ConditionIntegerRange`
- `ConditionDoubleRange`, `ConditionDoubleDualRange`

This is what backs `EventFilter`, `ParticleFilter`, `JetFilter`. A typical filter block looks like:

```ini
Filter0:Name                = PiP
Filter0:nConditions         = 2
Filter0:Condition0:Type     = DoubleRange
Filter0:Condition0:Subtype  = PT
Filter0:Condition0:Minimum  = 0.0
Filter0:Condition0:Maximum  = 1000000.0
Filter0:Condition1:Type     = Integer
Filter0:Condition1:Subtype  = PDG
Filter0:Condition1:Minimum  = 211
Filter0:Condition1:Maximum  = 211
```

### 4.5 Exceptions, logging, utilities

`Exception` is the root of a hierarchy: `NullPointerException`, `IndexOutOfBoundException`, `MemoryException`, `FactoryException`, `FileException`, `EndOfFileException`, `ConfigurationException`, `HistogramException`, `IncompatibleHistogramException`, `UnknownClassException`, `ResourceNotFoundException`, `FilterException`. `MATH::MathException` is parallel (DomainException, DivByZero, …).

`MessageLogger` defines severity levels (`kDebug0..2, kInfo, kWarning, kError, kFatal`) with each Object holding a `_minimumReportLevel` knob, set in the .ini via `Severity = Info`.

Other Base/ utilities: `Accountant` (counters), `Aliases` (`String = TString`, `Histogram1D = TH1` typedefs), `CmdlineParser`, `Crc32`, `EnvironmentVariables`, `Factory<T>` (object-pool template), `NameHelpers`, `PhysicsConstants`, `PrintHelpers`, `StreamHelpers`, `Table2D`, `Timer`, `SelectionGenerator`.

## 5. The Particles/ Module — Data Model & Event Loop

This is the core domain. It defines the physics objects and the per-event execution machinery.

### 5.1 Data classes

- **`Particle`** — 4-momentum (px, py, pz, E), 4-position (x, y, z, t), pointer to `ParticleType`, parent/children pointers (decay tree), live status, optional truth-particle pointer (for performance studies). Methods: boost, decay, distance.
- **`ParticleType`** — species metadata: PDG code, mass, charge, spin, isospin, baryon number, quark/lepton content, decay-mode list, enable flags. Predicates: `isProton()`, `isPion()`, `isLambda()`, …
- **`ParticleDecayMode`** — branching ratio + child PDGs, resolved to `ParticleType*` after DB load
- **`ParticleDb`** — singleton-style registry of all `ParticleType`s; lookup by PDG, by name; ASCII / XML import/export; bulk enable/disable by lifetime, weak-decay status, etc.
- **`ParticleDigit`** — bit-packed compact form for histograms
- **`Event`** — owns a `vector<Particle*>`, plus pre-allocated arrays of multiplicity / energy / charge / strangeness / baryon-number / pT-sums / spherocity, indexed by event-class bin (matches `EventFilter` count)

### 5.2 Filters

- `EventFilter` (← `Filter<Event>`) — multiplicity, charge, strangeness, baryon, energy, spherocity bins
- `ParticleFilter` (← `Filter<Particle>`) — pT/p/E, eta/y/phi, PDG, charge, strangeness, charm, beauty, live, enabled, efficiency
- `ParticlePairFilter` — for pair analyzers
- `ParticleTypeFilter` — selects on metadata only

### 5.3 Driver tasks

- **`EventFilterCreator`** — reads the `EventFilterCreator:EventFilter:FilterN:*` blocks, builds `EventFilter` objects, registers them by name
- **`ParticleFilterCreator`** — same for particle filters
- **`FilterCreator`** — combines both (also `JetFilter`)
- **`ParticleTypeTask` / `ParticleDbTask`** — loads a `ParticleDb` from disk; configurable to disable neutrals, weak decays, specific species
- **`EventIterator`** — the event loop. Configured with `nEventsRequested` and `nEventsReport`; on `execute()` it loops up to N times, calling `execute()` on each child per iteration. Children are typically (a) one generator + (b) one or more analyzers.
- **`FileIterator`** — analogous for post-processing: loops over input ROOT files instead of events

### 5.4 Analyzer base classes

- `EventProcessor` — abstract base inheriting `Task`, hosts filter/DB/stream lists and histogram lifecycle
- `EventProcessorSingle<H, DH>` — template for single-particle analyzers; owns a `H` and a `DH` (derived) histogram group
- `EventProcessorPair<H, DH, BfH, SH>` — template for pair analyzers; owns 4 histogram groups (primary pair, derived, balance-function, single-particle-derived)

`HistogramGroup` is the base class for any module-specific histogram bundle (1D/2D/3D `TH1`/`TH2`/`TH3` pointers + scaling/filling methods). `ManagedObjects` / `ManagedObject` wrap ROOT lifecycle (write to TFile, owned vs. borrowed).

### 5.5 Auxiliary tasks

- `ParticleDecayer` / `ParticleDecayerTask` — decays unstable particles using `ParticleDb` branching ratios
- `ParticleSmearer` / `EventSmearer` — apply detector smearing to particles / events

## 10. End-to-End: How a Pythia pp Job Runs

Reading `projects/Pythia/pp_13.7TeV/RunAna.ini` (which configures three top-level tasks: `RunAnalysis`, `RunDerived`, `RunBf`):

### Step 1. Bootstrap
`./CAP RunAnalysis Pythia/pp_13.7TeV RunAna.ini`
→ `CAP.cxx` parses argv, loads `RunAna.ini` into a `Configuration` (a flat list of `Property`s), and creates a single dummy "creator" Task that calls `createTask(nullptr, "RunAnalysis", config)`.

### Step 2. Tree construction (recursion)
`createTask` reads `RunAnalysis:TaskClassName = CAP::RunAnalysis`, calls `TClass::GetClass("CAP::RunAnalysis")->New()`, then for each declared subtask repeats. The resulting tree:

```
RunAnalysis  (CAP::RunAnalysis)
├── ParticleTypeTask         loads DefaultDb particle DB
├── EventFilterCreator       builds EventFilter "ALL", "MB" (multiplicity 0–100k)
├── ParticleFilterCreator    builds ParticleFilter "PiP","KP","PP","PiM","KM","PM","ALL"
└── EventIterator            10,000,000 events; report every 1000
    ├── PythiaEventGenerator √s=13 TeV, pp 2212+2212, 22 Pythia options (SoftQCD,
    │                        ColourReconnection, ParticleDecays:limitTau0=on, etc.)
    ├── ParticleSingleAnalyzer → SingleGen.root  (yields, pT, y, φ histograms)
    ├── ParticlePairAnalyzer  → PairGen.root     (Δφ-Δy correlations)
    └── ParticlePair3DAnalyzer → Pair3DGen.root  (Qinv, side/out/long)
```

### Step 3. Lifecycle execution

```cpp
mainTask->initialize();   // recursively initializes all children
mainTask->execute();      // RunAnalysis::execute() typically just calls
                          //    ParticleTypeTask once,
                          //    Filter creators once,
                          //    EventIterator once.
                          // EventIterator::execute() loops 10M times,
                          // each iteration:
                          //    PythiaEventGenerator::execute() → fills Event
                          //    ParticleSingleAnalyzer::execute() → fills histos
                          //    ParticlePairAnalyzer::execute()
                          //    ParticlePair3DAnalyzer::execute()
mainTask->finalize();     // each Analyzer writes its .root file via ManagedObjects
```

Inside an analyzer's `execute()`, a typical pattern is:

```cpp
for (auto * evtFilter : _eventFilters)
   if (evtFilter->accept(*event))
      for (auto * pfilter : _particleFilters)
         for (auto * particle : event->particles())
             if (pfilter->accept(*particle))
                _histos[bin]->fill(*particle);
```

For pair analyzers, an inner double-loop over particles is added.

### Step 4. Post-processing (RunDerived)

`./CAP RunDerived …` runs a different top-level task that uses `FileIterator` instead of `EventIterator` — it reads the histogram files from step 3 and runs `Calculator` tasks (`ParticlePair3DCalculator`, `ParticlePairCalculator`, `ParticleSingleCalculator`) to derive cumulants, correlation functions, etc., writing `*Derived.root` files.

### Step 5. Balance functions (RunBf)

`./CAP RunBf …` reads the derived files and runs `ParticlePair3DBfCalculator` → `Pair3DGenBf.root`.

### Step 6. Plots / tables

User runs ROOT macros from `src/Macros/` (e.g. `root PlotBF.C`) to load the result files and produce PDFs.

## 11. Cross-cutting Conventions

- **Naming**: `*Task` = a non-iterating Task; `*Analyzer` = histograms-during-event-loop; `*Calculator` = post-processing; `*Generator` = event source; `*Reader` = file-based event source; `*Iterator` = loop driver; `*Creator` = builds and registers filters/DBs; `*Histos` = `HistogramGroup` subclass; `*DerivedHistos` = post-processed histograms; `*BfHistos` = balance-function histograms.
- **Owner flags**: every resource (filter, DB, stream) listed in a Task's config has an `Owner0/1/…` flag — `1` means this task owns and will free the object; `0` means it borrows from a parent / sibling. This is critical for the lifetime model.
- **Severity**: every Task accepts `Severity = Debug | Info | Warning | Error`, controlling `MessageLogger` verbosity.
- **`HistogramsImportPath`/`Default`**: when `Default` is used, the task falls back to env vars `CAP_HISTOS_IMPORT_PATH` / `CAP_HISTOS_EXPORT_PATH` set by `SetupCAP.sh` (and overridden per-job by argv to `CAP.cxx`).
- **Object pooling**: hot loops (per particle, per pair) use `Factory<T>` to recycle preallocated objects rather than malloc each event.
- **ROOT dictionaries**: every module has a `*LinkDef.h` that lists the classes registered with ROOT for reflection (so `TClass::GetClass("CAP::Foo")` works and so the classes can be streamed to TFile).
